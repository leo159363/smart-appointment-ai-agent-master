# 项目包装参考

## 总体叙事

将项目定位为：

**面向家教/教培机构的多 Agent + RAG 智能课程咨询与排课 MVP，并以测试基线和演示验收保证领域迁移质量。**

核心一句话：

"系统通过中心化 Multi-Agent 编排，将课程咨询、老师匹配、试听课预约、正式排课、学习需求分析和课程知识库问答串成一条可演示的智能业务链路，并通过 pytest 基线、页面/API 验收、旧文案扫描和演示数据重置控制迁移风险。"

## 包装方向 1：测试开发项目

可以强调：

- 迁移前建立基线。
- 每阶段只改允许范围。
- 使用 `import app`、pytest collect-only、目标单测、页面/API 手工验收验证。
- 将失败分为 A/B/C/D/E。
- 通过关键词扫描定位外显残留。

不要声称：

- 全量测试已全部通过。
- 所有失败已修复。

## 包装方向 2：Agent 应用项目

可以强调：

- 任务分类 Agent 负责意图路由。
- 咨询 Agent 负责课程知识问答。
- 预约/排课 Agent 负责信息抽取、缺失追问、老师匹配和成功提示。
- 学习需求分析 Agent 负责偏好分析和回访提醒。

边界：

- 当前是中心化编排，不是复杂多智能体自治系统。

## 包装方向 3：RAG 知识库项目

可以强调：

- 当前使用 SQLite 存储课程知识文档。
- 使用 Embedding 和 FAISS 构建语义索引。
- 覆盖课程体系、收费规则、老师介绍、试听/排课规则、课时包和教学质量。
- 后续可接入独立 Modular RAG MCP Server 做检索、引用、Trace 和评估。

边界：

- 当前 RAG 是轻量内置方案。
- 不要写成已完成独立 RAG/MCP 服务集成。

## 包装方向 4：FastAPI 后端项目

可以强调：

- Web routes 和 API routes 分层。
- `/chat/stream` 流式响应。
- Service 层封装知识库、老师、预约和学习需求数据访问。
- SQLite 演示数据可重建。

## 包装强度分级

| 级别 | 写法 |
|---|---|
| 保守 | "完成家教场景 MVP 迁移，具备多 Agent、RAG 知识库和基础测试基线。" |
| 中等 | "围绕测试开发目标，建立迁移基线、缺陷分类和页面/API/数据验收流程。" |
| 偏 Agent/RAG | "基于中心化多 Agent 编排和轻量 RAG 课程知识库，构建可演示的智能咨询与排课链路。" |
| 未来增强 | "规划接入独立 RAG/MCP 知识层和自动化评估体系，进一步提升检索可信度和回归稳定性。" |

- AI 家教培训机构智能咨询与排课系统
